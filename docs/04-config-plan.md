# 04 配置管理文档

## 1. 配置项范围
- README
- 项目启动说明
- WBS 文档
- 进度计划文档
- 配置管理文档
- 总结报告


## 2. 分支策略
- main: 受保护分支，仅允许 Merge Request 合并。
- dev: 日常开发集成分支。
- feature-*: 功能开发分支。

## 3. 命名约定
### 文档/文件命名：
采用 序号-功能模块.后缀 格式。
### 分支命名：
main ：生产主分支，代码必须稳定，受保护。
dev ：开发集成分支，日常开发合并的目标。
feature-功能名：新功能开发分支。。
### Git 提交规范 ：
作者设置：严格遵守指导书，Git 用户名必须设置为本人姓名拼音全称（如 zhangsan），以便追溯个人贡献。
信息格式：[类型] 简述
[Add]：新增文件或功能。
[Update]：修改现有代码。
[Docs]：仅修改文档。
[Merge]：分支合并。
## 4. 合并规则
### 合并责任人：
组长：负责审核代码并合并到 main 和 dev 分支，确保代码质量。
普通成员：仅在自己的 feature-* 分支开发，严禁直接推送到 main。
### 合并流程：
开发阶段：成员从 dev 拉取最新代码，创建 feature-* 分支进行开发。
Pull Request (PR)：开发完成后，向 dev 分支发起合并请求（或直接合并，视实验复杂度而定）。
集成测试：所有功能合并至 dev 进行集成。
发布阶段：实验结束/项目完成时，将 dev 合并至 main。
### 冲突处理：
原则：谁开发谁解决。如果在 PR 时发现冲突，开发者需先在本地解决冲突，测试通过后再合并。
强制要求：必须保留冲突解决的 Git 提交记录。

## 5. 版本留痕
### 阶段性版本保留：
里程碑：每个主要交付物完成时打一个标签。
M1：完成 WBS 和进度计划（提交至 dev）。
M2：完成系统设计与配置管理方案。
M3：完成总结报告（最终合并至 main）。
