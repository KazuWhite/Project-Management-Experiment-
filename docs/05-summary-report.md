### 1. 本组项目组织情况
我们小组围绕“校园活动报名与签到管理系统”项目，设定了明确的角色分工。罗成瑞鑫负责仓库初始化、WBS 分解及进度计划；赖瑞林负责项目章程、整体协调以及README的撰写；李卓卿负责配置管理与分支策略。整个实验过程中，我们通过 GitHub 进行远程协作，所有成员均使用自己的姓名拼音作为 Git 提交者标识，保证了提交记录的清晰可追溯。

### 2. WBS 分解思路
我们的 WBS 严格遵循“以交付物为导向”的原则，将项目成果分解为六大领域：项目管理成果、系统需求与设计成果、活动发布与管理功能成果、学生报名与审核功能成果、签到与统计功能成果、测试与验收成果。每个领域下再细分至具体文档或产出物，共形成三级结构。分解过程中我们特别避免使用“开会”“讨论”等动作性描述，确保每一项都是可查验的实质性成果。

### 3. 进度安排依据
基于 WBS 提取关键活动后，我们建立了简化的进度计划表。关键路径为：仓库初始化（A1）→ 配置管理方案（A5）→ 冲突解决（A6）→ 总结报告（A7）→ 最终合并（A8）。各活动的前置关系主要依据文档间的依赖逻辑设定，如章程完成后才能完善 README，WBS 完成后才能制定进度计划。所有活动责任人均分配至具体成员，工期均设为 1 天，符合实验课时要求。

### 4. 配置管理与协作过程
我们为仓库设定了 main、dev 及 feature-* 三级分支策略。所有功能文档均在个人分支上开发，经自测后合并至 dev 分支；最终由 dev 统一合并至 main，确保 main 分支始终处于受控状态。协作期间，我们使用 GitHub 的 Pull Request 功能（或直接合并）进行审查，并通过 VS Code 的 Git Graph 插件直观查看分支演进。提交信息采用“docs: ...”的统一格式，便于追溯。

### 5. 冲突解决过程记录
本次实验我们通过冲突示例文件 conflict.md 刻意制造了一次真实的合并冲突。
- **冲突起因**：成员罗成瑞鑫在 feature-conflict-A 分支将“当前结论”修改为“所有功能分支必须先通过 Pull Request 合并到 dev，并通过代码审查后才能合并到 main”；成员李卓卿在 feature-conflict-B 分支将同一行修改为“为简化流程，同意在实验环境下直接合并功能分支到 dev 并跳过审查，但最终合并到 main 必须由项目经理确认”。两人从不同的角度对合并规则进行描述，导致合并时同一位置内容冲突。
- **冲突识别**：当执行 git merge feature-conflict-B 时，Git 自动检测到 docs/conflict.md 存在内容冲突，标记为 CONFLICT (content)，并在文件中用 `<<<<<<<`、`=======`、`>>>>>>>` 标识出双方差异。
- **解决方式**：我们在 VS Code 中打开冲突文件，利用其内置的合并编辑器，经过小组讨论后，手动将冲突部分统一为“各功能分支在本地自测后可直接合并至 dev 分支，但在合并至 main 分支前必须经过全组复查并由配置管理员执行合并操作”，确保了一致性。随后执行 git add 和 git commit 完成手动合并。
- **最终结果**：冲突成功解决，dev 分支上留下一笔清晰的合并提交 conflict: 手动解决合并冲突，统一当前结论，完整的冲突与解决过程被永久记录在版本库中。

### 6. 经验与问题
**收获**：
- 深入理解了 Git 分支模型与合并冲突的实际处理，学会了通过 VS Code 工具解决冲突。
- 掌握了从 WBS 到进度计划的项目管理基本流程，认识到结果导向的分解对项目管理的重要性。
- 体会了团队协作中配置管理的关键作用，尤其是分支策略和提交规范的必要性。

**遇到的问题**：
- 初期因编辑器未保存文件，导致提交时未发现变更，后来意识到修改后必须保存。
- 使用不同终端环境时出现 SSL 证书问题，通过临时关闭 SSL 验证解决，并了解到 Windows 下 Git 配置的局限性。
- 团队成员在首次推送时也遇到了身份识别问题，通过重新设置 user.name 和 user.email 解决。

**改进建议**：
- 在开始协作前统一所有成员的 Git 配置和终端环境，可以减少重复性错误。
- 可以设定更明确的提交信息模板，进一步提高可追溯性。
