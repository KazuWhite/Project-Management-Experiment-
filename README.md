# 校园活动报名与签到管理系统

> 软件项目管理课程综合实验 · 软件工程23级 · 三人协作项目

---
## 1.项目背景与总体目标

本项目为软件项目管理课程综合实验，以“校园活动报名与签到管理系统”为统一业务背景。项目**不以完整业务系统开发为目标**，而是以软件项目管理方法实践为核心，重点训练项目规划、组织与控制能力。

本实验强调“以过程为导向”，关注从需求分析到成果交付全过程中的管理规范性与可追溯性，而非系统功能实现的复杂程度。

本实验围绕软件项目管理的四个关键领域展开，包括：范围管理、进度管理、配置管理以及团队协作能力。

---

### 1.1 范围管理

范围管理的核心在于明确项目“做什么”与“不做什么”，避免范围蔓延。

#### （1）功能拆解

围绕系统核心业务进行划分，主要包括：

- 活动管理
- 报名管理
- 审核与签到
- 统计分析

#### （2）三级 WBS 分解

构建面向交付物的工作分解结构（WBS）：

- **一级**：项目整体（校园活动管理系统）
- **二级**：功能模块（活动管理、报名管理等）
- **三级**：具体工作项（接口设计、数据结构定义、文档编写等）

#### （3）面向交付物原则

每个WBS节点必须满足：

- 可交付（如文档、代码模块）
- 可验证
- 可度量

避免使用模糊描述（如“优化功能”、“完善系统”等）。

#### （4）范围边界控制

明确实验范围：

- 包含：WBS文档、进度计划、Git管理、协作过程  
- 不包含：复杂UI设计、完整系统部署等  

---

### 1.2 进度管理

进度管理的目标是合理安排任务顺序与时间，提高项目执行效率。

#### （1）关键活动提取

从WBS中识别关键任务，例如：

- 项目初始化
- WBS设计
- 进度计划制定
- Git仓库搭建
- 分支开发
- 冲突解决

#### （2）任务依赖关系

明确任务之间的先后关系：

- WBS → 进度计划  
- 仓库初始化 → 分支开发  
- 分支开发 → 合并与冲突处理  

#### （3）进度计划制定

采用简化形式（表格或甘特图）进行进程计划的制定

#### （4）关键路径意识

识别关键任务链，避免关键路径延误影响整体进度。

---

### 1.3 配置管理

配置管理的核心是确保项目成果的可控性与可追溯性。

#### （1）版本库建立

- 使用 Git 初始化项目仓库
- 提交项目基础结构

#### （2）分支管理策略

采用标准分支模型：

- `main`：主分支（稳定版本）
- `dev`：开发主分支
- `feature-*`：功能分支（个人开发）

#### （3）版本追溯

- 所有修改均需记录
- 支持历史版本回溯与问题定位

---

### 1.4 团队协作与冲突管理

该部分重点训练团队开发中的实际问题处理能力。

#### （1）多人分支协作

- 每位成员基于独立分支开发
- 通过合并请求进行代码整合

#### （2）冲突制造

- 多人修改同一文件同一位置
- 人为触发Git冲突

#### （3）冲突解决

掌握冲突处理方法

## 5. 协作约定

提交规范：提交信息使用 docs: <具体修改内容> 格式，如 docs: 完成三级WBS分解；冲突解决提交信息以 conflict: 开头。

合并规则：个人分支合并到 dev 前需先推送至远程，由配置管理员在本地拉取并合并；dev 合并到 main 须经全组复查后由配置管理员执行。

身份要求：每位成员严格使用本人姓名拼音设置 user.name，提交前用 git log -1 --format='%an %ae' 自查。

冲突处理：合并时若发生冲突，禁止直接推送，需在本地拉取最新 dev 后手工解决，确认无误后再推送。

---


## 6. 分支管理策略

本项目采用 Git Flow 简化模型：

feature → dev → main

规则如下：
- 所有开发必须基于 feature 分支进行
- 完成后合并至 dev 分支进行集成
- dev 分支稳定后合并至 main 作为最终交付
- main 分支禁止直接提交

冲突处理：
- 冲突由配置管理员统一协调解决
- 所有冲突需记录在 conflict.md 中

## 8. 提交历史说明

所有成员均在 feature-* 分支上完成各自的文档初稿，并合并到 dev 进行集成。冲突文件 conflict.md 由两位成员分别修改同一行，在合并到 dev 时触发冲突，随后通过VS Code手动解决并提交。最终 dev 分支的内容被完整合并到 main，形成本仓库的最终提交版本。完整的提交记录和协作痕迹可在 commits 页面和 Git 图表中查看。
